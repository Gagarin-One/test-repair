# Ключевые архитектурные решения

## 1. Стек технологий
- **Фронтенд**: React + TypeScript, Vite, FSD архитектура
- **Бэкенд**: Node.js + Express + TypeScript
- **База данных**: PostgreSQL
- **Контейнеризация**: Docker Compose 

## 2. Feature-Sliced Design на фронтенде
Выбрал FSD для лучшей масштабируемости и поддержки кода. Разделение на слои (app, pages, features, entities, shared) позволяет четко разделить ответственность и упрощает навигацию по проекту.

## 3. Защита от гонок (race condition)
Использовал транзакции с `SELECT ... FOR UPDATE` в PostgreSQL. Это блокирует строку на время обновления, гарантируя, что только один запрос сможет изменить статус заявки. Остальные получают 409 Conflict.

Создан bash-скрипт `race_test.sh`, который отправляет 10 параллельных запросов и проверяет, что только один проходит.

## 4. Аутентификация
Для тестового задания выбрал упрощенную аутентификацию через React Context с предопределенными пользователями. Это позволило быстро реализовать разделение ролей без сложной бэкенд-логики.

## 5. API дизайн
RESTful API с четким разделением эндпоинтов:
- `POST /api/requests` - создание заявки (публичный)
- `GET /api/requests` - список заявок с фильтрацией
- `PATCH /api/requests/:id` - обновление заявки
- `POST /api/requests/:id/take` - специальный эндпоинт для взятия в работу (с защитой)

## 6. Валидация данных
Использовал Zod на бэкенде для валидации входящих данных. Это обеспечивает типобезопасность и понятные сообщения об ошибках.

## 7. Структура базы данных
Минималистичная схема с двумя таблицами (users и requests) и внешними ключами. Статусы заявок ограничены через CHECK constraint.

## 8. Обработка ошибок
На бэкенде возвращаются соответствующие HTTP статусы (400, 404, 409, 500). На фронтенде пользователь видит понятные сообщения об ошибках.
